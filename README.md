## Features

- **Profile** вЂ” personal info entity, served via REST API
- **Skills** вЂ” skill entries organized into groups
- **Education** вЂ” education history entries
- **Images** вЂ” media management with dual-storage support:
  - Local filesystem upload via AdminJS panel
  - Optional one-click migration to S3 вЂ” upload, get public URL, update DB, remove local file automatically
- **SEO** вЂ” fully dynamic SEO managed from the admin panel (see below)
- **Admin Panel** вЂ” full CRUD via AdminJS at `/admin`
- **i18n ready** вЂ” Russian locale included, easy to extend
- **Dark/light theme** вЂ” saved in cookie, applied before first render (no flash)

## SEO

All SEO settings are stored in the database and managed via the admin panel вЂ” no code changes needed.

### What's covered

| Feature | Details |
|---|---|
| Meta tags | `title`, `description`, `keywords`, `author`, `robots`, `theme-color` |
| Open Graph | `og:title`, `og:description`, `og:image` (1200Г—630), `og:type`, `og:locale`, `og:site_name`, `og:url` |
| Twitter Card | `summary_large_image`, `twitter:site`, `twitter:creator` |
| Schema.org JSON-LD | `Person` вЂ” name, jobTitle, url, image, sameAs (GitHub, Telegram, email) |
| Canonical URL | Set from DB, injected via Next.js `alternates` |
| Favicon set | `.ico`, `.svg`, `96Г—96 png`, `apple-touch-icon 180Г—180` |
| Web App Manifest | PWA-ready вЂ” 192Г—192 and 512Г—512 maskable icons |
| robots.txt | Auto-generated by Next.js, points to sitemap |
| sitemap.xml | Auto-generated by Next.js, `changeFrequency: monthly` |

### How it works

The backend exposes `GET /seo-meta` which returns all SEO fields. Next.js fetches it server-side at build/request time (revalidated every hour) and injects everything into `<head>` automatically.

JSON-LD is rendered directly in `layout.tsx`:

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Person",
  "name": "...",
  "jobTitle": "...",
  ...
}
</script>
```

## Getting Started

### Prerequisites

- [Docker](https://www.docker.com/) and Docker Compose

### Run locally

```bash
git clone <repo-url>
cd tee-su
cp backend/.env.example backend/.env
# fill in backend/.env
docker compose up
```

| Service | URL |
|---|---|
| Frontend | <http://localhost:3000> |
| Backend API | <http://localhost:4000> |
| Admin Panel | <http://localhost:4000/admin> |

```env
S3_REGION=
S3_PUBLIC_URL=
```

## S3 Image Upload

Images can be stored locally or pushed to any S3-compatible storage (AWS S3, Beget, MinIO, etc.).

In the admin panel, when creating or editing an image, check **Save To S3**. After the file is saved locally, the backend will automatically:

1. Read the file from local storage
2. Upload it to the configured S3 bucket
3. Retrieve the public URL
4. Update the database record
5. Delete the local file

All steps are logged via NestJS Logger for easy debugging.

## Production Deployment

### Server requirements

- 1 CPU / 1 GB RAM minimum (2 GB RAM recommended)
- 20 GB disk
- Ubuntu 22.04+
- Docker installed
- Domain A-records pointed to the server IP

> **Note:** On 1 GB RAM servers, add swap before deploying:
>
> ```bash
> fallocate -l 2G /swapfile && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile
> echo '/swapfile none swap sw 0 0' >> /etc/fstab
> ```

### Step 1 вЂ” Prepare the server

```bash
curl -fsSL https://get.docker.com | sh
```

### Step 2 вЂ” Clone and configure

```bash

cd /var/www/tee-su
cp backend/.env.example backend/.env

```

### Step 3 вЂ” First deploy

```bash

```

Verify the site is working:

| Service | URL |
|---|---|
| Frontend | <http://yourdomain.com> |
| Admin Panel | <http://yourdomain.com/admin> |

### Step 4 вЂ” Enable HTTPS

> Run once, after the site is accessible over HTTP.

```bash
apt install -y certbot
bash ssl-init.sh
```

The script:

1. Stops nginx temporarily to free port 80
2. Obtains a Let's Encrypt certificate for your domain
3. Restarts nginx вЂ” HTTP redirects to HTTPS automatically

---

### Step 5 вЂ” SSL auto-renewal

```bash
crontab -e
```

Add:

```
0 3 * * * certbot renew --quiet && docker compose -f /var/www/tee-su/docker-compose.prod.yml restart nginx
```

---

### Step 6 вЂ” Auto-deploy via GitHub Actions

On every `git push main`, GitHub Actions SSHs into the server and runs `deploy.sh` automatically.

#### Setup

1. Generate a deploy SSH key on the server:

```bash
ssh-keygen -t ed25519 -C "github-actions-deploy" -f ~/.ssh/github_deploy -N ""
cat ~/.ssh/github_deploy.pub >> ~/.ssh/authorized_keys
```

2. Add 4 secrets to the GitHub repo (`Settings в†’ Secrets and variables в†’ Actions`):

| Secret | Value |
|---|---|
| `SSH_HOST` | your domain or server IP |
| `SSH_USER` | `root` (or your user) |
| `SSH_PRIVATE_KEY` | contents of `~/.ssh/github_deploy` |
| `PROJECT_PATH` | `/var/www/tee-su` |

The workflow file is at `.github/workflows/deploy.yml`.

## Known Issues & Solutions

### AdminJS components.bundle.js вЂ” 404 in production

**Problem:** In production Docker, AdminJS shows `Component "ComponentX" has not been bundled` error. The bundle file exists at runtime but Express returns 404.

**Root cause:** Two issues combined:

1. Express `send` module ignores dotfile directories by default вЂ” returns 404 for any path containing a component starting with `.`. The default AdminJS tmp dir `.adminjs` triggered this.
2. `@adminjs/upload` registers components via `AdminJS.bundle()` only when `uploadFeature()` is called вЂ” not at import time, causing a race condition with `initializeAdmin()`.

**Solution:**

- Pre-generate the bundle during Docker build via `backend/scripts/pre-bundle.js`
- Set `ADMIN_JS_TMP_DIR=adminjs` (no leading dot) to bypass Express dotfile restriction
- Set `ADMIN_JS_SKIP_BUNDLE=true` to prevent runtime re-bundling

See: `backend/scripts/pre-bundle.js`, `backend/Dockerfile`, `docker-compose.prod.yml`

---

### Next.js Image Optimization вЂ” 400 Bad Request for external URLs

**Problem:** `/_next/image?url=https://...` returns 400 in production.

**Root cause:** `next.config.ts` was not copied to the runner stage in the frontend Dockerfile.

**Solution:** Added `COPY --from=builder /app/next.config.ts ./next.config.ts` to the runner stage in `frontend/Dockerfile`.

---

### Docker build OOM on 1 GB RAM

**Problem:** `nest build` crashes with `JavaScript heap out of memory` during Docker build.

**Solution:**

- Add 2 GB swap (see server requirements above)
- Switch NestJS compiler to SWC: set `"builder": "swc"` in `nest-cli.json`
- Install SWC: `npm install --save-dev @swc/cli @swc/core`

SWC is a Rust-based TypeScript compiler вЂ” 20x faster than `tsc` and uses ~10x less memory.
